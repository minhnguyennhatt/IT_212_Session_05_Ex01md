# Bài làm: Phân tích & Lựa chọn Prompt tối ưu

## 1. Đáp án lựa chọn

**Chọn phương án B.**

Phương án B là prompt tối ưu nhất vì nó mô tả rõ vai trò, nhiệm vụ, ràng buộc kỹ thuật, quy tắc nghiệp vụ, quy trình suy luận từng bước và yêu cầu đầu ra cuối cùng. Với bài toán tính thuế TNCN lũy tiến, nếu prompt không đủ chi tiết thì AI rất dễ sinh code sai ở phần ranh giới bậc thuế, sai cách tính lũy tiến từng phần hoặc sai do dùng kiểu dữ liệu `double`.

---

## 2. Vì sao phương án B tối ưu nhất?

Phương án B tốt nhất vì áp dụng tốt kỹ thuật **Chain-of-thought prompting** ở mức yêu cầu AI phân tích theo từng bước trước khi sinh code.

Cụ thể, phương án B có các điểm mạnh sau:

### 2.1. Có vai trò rõ ràng

Prompt yêu cầu AI đóng vai:

> Java Developer chuyên nghiệp kiêm chuyên gia tài chính

Điều này giúp AI hiểu rằng bài toán không chỉ là viết code Java thông thường, mà còn liên quan đến nghiệp vụ tài chính, tính tiền, thuế và độ chính xác số học.

---

### 2.2. Có nhiệm vụ cụ thể

Prompt nêu rõ nhiệm vụ:

> Viết một class Java tính thuế thu nhập cá nhân lũy tiến dựa trên thu nhập chịu thuế.

Nhiệm vụ này rõ hơn nhiều so với câu “hãy tối ưu nhất” ở phương án A. AI biết chính xác cần sinh ra cái gì: một class Java tính thuế.

---

### 2.3. Có ràng buộc kỹ thuật rõ ràng

Phương án B yêu cầu:

- Sử dụng **Java 17**
- Sử dụng **BigDecimal**
- Tránh sai số làm tròn tiền tệ

Đây là điểm rất quan trọng. Trong bài toán tiền tệ, nếu dùng `double` hoặc `float`, có thể phát sinh sai số thập phân. `BigDecimal` phù hợp hơn khi tính toán tiền lương, bảo hiểm và thuế.

---

### 2.4. Có dữ liệu nghiệp vụ cụ thể

Prompt B liệt kê đầy đủ các bậc thuế:

| Bậc |         Thu nhập tính thuế | Thuế suất |
| --- | -------------------------: | --------: |
| 1   |                Đến 5 triệu |        5% |
| 2   |  Trên 5 triệu đến 10 triệu |       10% |
| 3   | Trên 10 triệu đến 18 triệu |       15% |
| 4   | Trên 18 triệu đến 32 triệu |       20% |
| 5   | Trên 32 triệu đến 52 triệu |       25% |
| 6   | Trên 52 triệu đến 80 triệu |       30% |
| 7   |              Trên 80 triệu |       35% |

Việc nêu rõ bậc thuế giúp AI tránh tự suy đoán sai quy định hoặc bỏ sót bậc thuế.

---

### 2.5. Có quy trình Chain-of-thought rõ ràng

Prompt B yêu cầu AI không viết code ngay mà phải đi theo các bước:

1. Phân tích cách xác định thu nhập tính thuế.
2. Liệt kê công thức tính thuế từng bậc.
3. Chạy thử bằng tay với ví dụ cụ thể.
4. Cuối cùng mới sinh mã nguồn Java.

Quy trình này giúp giảm nguy cơ AI sinh code sai logic. Đặc biệt với bài toán lũy tiến từng phần, việc dry-run trước khi code là rất quan trọng để kiểm tra biên tính toán.

---

## 3. Phân tích theo cấu trúc 5 thành phần của Prompt

Phương án B đáp ứng tốt 5 thành phần của một prompt hiệu quả:

### 3.1. Vai trò

Prompt có vai trò rõ:

> Java Developer chuyên nghiệp kiêm chuyên gia tài chính

Vai trò này phù hợp với bài toán vừa cần kỹ năng lập trình, vừa cần hiểu nghiệp vụ tài chính.

### 3.2. Bối cảnh

Prompt cung cấp bối cảnh là bài toán tính thuế TNCN lũy tiến trong hệ thống quản lý nhân sự.

Điều này giúp AI hiểu đây là logic nghiệp vụ nghiêm túc, cần độ chính xác cao.

### 3.3. Nhiệm vụ

Nhiệm vụ là viết class Java tính thuế TNCN lũy tiến.

Nhiệm vụ rõ ràng, không mơ hồ.

### 3.4. Ràng buộc

Prompt có các ràng buộc quan trọng:

- Java 17
- Dùng `BigDecimal`
- Có các khoản giảm trừ:
  - Giảm trừ bản thân: 11 triệu
  - Người phụ thuộc: 4.4 triệu/người
  - Bảo hiểm bắt buộc: 10.5% tổng thu nhập

- Áp dụng biểu thuế lũy tiến từng phần

Các ràng buộc này giúp AI kiểm soát tốt logic nghiệp vụ và kỹ thuật.

### 3.5. Định dạng đầu ra

Prompt yêu cầu đầu ra theo quy trình rõ:

- Phân tích
- Công thức
- Dry-run
- Code Java hoàn chỉnh

Nhờ đó, kết quả dễ kiểm tra, dễ đánh giá và giảm nguy cơ sai logic.

## 4. Lý do loại trừ các phương án còn lại

## 4.1. Loại phương án A

Phương án A:

> "Hãy viết hàm Java tính thuế thu nhập cá nhân lũy tiến theo biểu thuế Việt Nam hiện hành. Hãy tối ưu nhất."

Nhược điểm:

- Quá ngắn và mơ hồ.
- Không nêu rõ dùng kiểu dữ liệu nào.
- Không yêu cầu dùng `BigDecimal`.
- Không liệt kê các bậc thuế.
- Không yêu cầu phân tích trước khi viết code.
- Cụm “tối ưu nhất” không rõ tối ưu về hiệu năng, độ chính xác, khả năng bảo trì hay độ ngắn gọn.
- AI có thể tự suy đoán biểu thuế, dẫn đến sai logic hoặc sai ranh giới bậc thuế.

Vì vậy, phương án A không phù hợp với bài toán cần độ chính xác cao như tính thuế.

---

## 4.2. Loại phương án C

Phương án C:

> "Hãy đóng vai trò Senior Developer. Hãy viết một class Java tính thuế TNCN lũy tiến sử dụng Java Stream API song song để tính toán cực nhanh các bậc thuế lũy tiến nhằm tối ưu hiệu năng."

Nhược điểm:

- Tập trung sai mục tiêu.
- Bài toán tính thuế TNCN chỉ có vài bậc thuế, không cần tối ưu hiệu năng bằng parallel stream.
- Dùng parallel stream có thể làm code phức tạp không cần thiết.
- Dễ gây khó kiểm soát thứ tự tính toán các bậc thuế.
- Không nhấn mạnh độ chính xác tiền tệ.
- Không yêu cầu `BigDecimal`.
- Không yêu cầu phân tích từng bước hoặc dry-run.

Với bài toán tài chính, điều quan trọng nhất là **đúng logic, đúng biên, dễ kiểm thử**, không phải “tính cực nhanh”.

# 5. Mã nguồn Java hoàn chỉnh

```java
import java.math.BigDecimal;
import java.math.RoundingMode;
import java.util.List;

public class PersonalIncomeTaxCalculator {

    private static final BigDecimal PERSONAL_DEDUCTION = new BigDecimal("11000000");
    private static final BigDecimal DEPENDENT_DEDUCTION = new BigDecimal("4400000");
    private static final BigDecimal INSURANCE_RATE = new BigDecimal("0.105");

    private static final List<TaxBracket> TAX_BRACKETS = List.of(
            new TaxBracket(new BigDecimal("5000000"), new BigDecimal("0.05")),
            new TaxBracket(new BigDecimal("10000000"), new BigDecimal("0.10")),
            new TaxBracket(new BigDecimal("18000000"), new BigDecimal("0.15")),
            new TaxBracket(new BigDecimal("32000000"), new BigDecimal("0.20")),
            new TaxBracket(new BigDecimal("52000000"), new BigDecimal("0.25")),
            new TaxBracket(new BigDecimal("80000000"), new BigDecimal("0.30")),
            new TaxBracket(null, new BigDecimal("0.35"))
    );

    public static void main(String[] args) {
        BigDecimal totalIncome = new BigDecimal("30000000");
        int numberOfDependents = 1;
        boolean includeInsurance = false;

        BigDecimal taxableIncome = calculateTaxableIncome(
                totalIncome,
                numberOfDependents,
                includeInsurance
        );

        BigDecimal tax = calculateProgressiveTax(taxableIncome);

        System.out.println("Tổng thu nhập: " + totalIncome + " VND");
        System.out.println("Số người phụ thuộc: " + numberOfDependents);
        System.out.println("Có trừ bảo hiểm: " + includeInsurance);
        System.out.println("Thu nhập tính thuế: " + taxableIncome + " VND");
        System.out.println("Thuế TNCN phải nộp: " + tax + " VND");
    }

    public static BigDecimal calculateTaxableIncome(
            BigDecimal totalIncome,
            int numberOfDependents,
            boolean includeInsurance
    ) {
        if (totalIncome == null || totalIncome.compareTo(BigDecimal.ZERO) < 0) {
            throw new IllegalArgumentException("Tổng thu nhập không được null hoặc âm");
        }

        if (numberOfDependents < 0) {
            throw new IllegalArgumentException("Số người phụ thuộc không được âm");
        }

        BigDecimal dependentDeduction = DEPENDENT_DEDUCTION.multiply(
                BigDecimal.valueOf(numberOfDependents)
        );

        BigDecimal insuranceDeduction = includeInsurance
                ? totalIncome.multiply(INSURANCE_RATE)
                : BigDecimal.ZERO;

        BigDecimal taxableIncome = totalIncome
                .subtract(PERSONAL_DEDUCTION)
                .subtract(dependentDeduction)
                .subtract(insuranceDeduction);

        if (taxableIncome.compareTo(BigDecimal.ZERO) <= 0) {
            return BigDecimal.ZERO;
        }

        return taxableIncome.setScale(0, RoundingMode.HALF_UP);
    }

    public static BigDecimal calculateProgressiveTax(BigDecimal taxableIncome) {
        if (taxableIncome == null || taxableIncome.compareTo(BigDecimal.ZERO) <= 0) {
            return BigDecimal.ZERO;
        }

        BigDecimal totalTax = BigDecimal.ZERO;
        BigDecimal previousLimit = BigDecimal.ZERO;

        for (TaxBracket bracket : TAX_BRACKETS) {
            BigDecimal currentLimit = bracket.upperLimit();
            BigDecimal rate = bracket.rate();

            BigDecimal taxableAmountInBracket;

            if (currentLimit == null) {
                taxableAmountInBracket = taxableIncome.subtract(previousLimit);
            } else {
                taxableAmountInBracket = taxableIncome.min(currentLimit).subtract(previousLimit);
            }

            if (taxableAmountInBracket.compareTo(BigDecimal.ZERO) > 0) {
                BigDecimal taxInBracket = taxableAmountInBracket.multiply(rate);
                totalTax = totalTax.add(taxInBracket);
            }

            if (currentLimit == null || taxableIncome.compareTo(currentLimit) <= 0) {
                break;
            }

            previousLimit = currentLimit;
        }

        return totalTax.setScale(0, RoundingMode.HALF_UP);
    }

    private record TaxBracket(BigDecimal upperLimit, BigDecimal rate) {
    }
}
```

# 6. Dry-run ví dụ trong đề

Ví dụ:

- Tổng thu nhập: 30.000.000 VND
- Số người phụ thuộc: 1
- Không tính bảo hiểm

Công thức:

```text
Thu nhập tính thuế = Tổng thu nhập
                   - Giảm trừ bản thân
                   - Giảm trừ người phụ thuộc
                   - Bảo hiểm
```

Thay số:

```text
Thu nhập tính thuế = 30.000.000 - 11.000.000 - 4.400.000 - 0
                   = 14.600.000 VND
```

Tính thuế lũy tiến từng phần:

```text
Bậc 1: 5.000.000 × 5% = 250.000
Bậc 2: 5.000.000 × 10% = 500.000
Bậc 3: 4.600.000 × 15% = 690.000
```

Tổng thuế:

```text
250.000 + 500.000 + 690.000 = 1.440.000 VND
```

Vậy với tổng thu nhập **30 triệu VND**, **1 người phụ thuộc**, **không bảo hiểm**, số thuế TNCN phải nộp là:

```text
1.440.000 VND
```

# 7. Kết luận

**Phương án B là lựa chọn tối ưu nhất** vì prompt có đầy đủ vai trò, bối cảnh, nhiệm vụ, ràng buộc và định dạng đầu ra. Đặc biệt, prompt yêu cầu AI phân tích từng bước trước khi viết code, giúp hạn chế lỗi logic trong bài toán tính thuế lũy tiến từng phần. Trong khi đó, phương án A quá mơ hồ, còn phương án C tập trung sai vào hiệu năng thay vì độ chính xác và khả năng kiểm chứng.
